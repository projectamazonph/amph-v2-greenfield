# STORY-043 — `/certificates/[hash]` Public View + `/pdf` Route

## Status

- **Story**: STORY-043
- **Sprint**: 9 — Certificates + Email
- **Points**: 1
- **Status**: Done (commit 020973b)

## Overview

Expose the certificate publicly via the verification hash. Two routes:

- **`GET /certificates/[hash]`** — HTML view (the public verification page).
  Anyone with the hash can view it. Shows a REVOKED badge if applicable.
- **`GET /certificates/[hash]/pdf`** — PDF download. Streams the
  certificate PDF generated by `RenderCertificatePdf` (STORY-042).

Both routes are PUBLIC. No authentication. The hash is the only secret
needed (and it's not a secret — it's just a stable identifier).

**Scope of this story:**
- `VerifyCertificate` use case (looks up by hash, returns cert + user + course)
- `src/app/certificates/[hash]/page.tsx` — HTML page
- `src/app/certificates/[hash]/pdf/route.ts` — API route
- `src/app/certificates/[hash]/not-found.tsx` — 404 view
- Wire `VerifyCertificate` into the DI container

**Out of scope:**
- QR code (deferred to 045 or later)
- Email notification on issuance — STORY-045
- Revocation badge styling tweaks — design pass later
- OG image / SEO niceties beyond the basics

## 1. Use Case — VerifyCertificate

```typescript
// src/usecases/VerifyCertificate.ts

export interface VerifyCertificateInput {
  verificationHash: string;
}

export type VerifyCertificateError =
  | { kind: "certificate_not_found" }
  | { kind: "user_not_found" }
  | { kind: "course_not_found" }
  | { kind: "invalid_hash_format" }
  | { kind: "db_error"; message: string };

export type VerifyCertificateResult = Result<
  {
    certificate: Certificate;
    user: { firstName: string; lastName: string };
    course: { title: string; tagline: string };
  },
  VerifyCertificateError
>;

export interface VerifyCertificateDeps {
  certificateRepo: ICertificateRepository;
  userRepo: UserRepository;
  courseRepo: CourseRepository;
}
```

**Flow:**
1. Validate hash format (64-char hex) → `invalid_hash_format`
2. Find certificate by hash → `certificate_not_found`
3. Find user → `user_not_found` (defensive — shouldn't happen but treat as 500)
4. Find course → `course_not_found` (same)
5. Return `{ certificate, user, course }` — including revoked certs

The user/course "not found" cases are degenerate (data integrity issue)
but we surface them explicitly rather than crashing. In practice the
foreign keys enforce this; if we get here, it's a real bug.

**Why not reuse `RenderCertificatePdf`?** That use case returns a
`Buffer` and calls a renderer. The HTML view doesn't need a buffer.
Different return shape, different orchestration. Two use cases.

## 2. Routes

### HTML view — `src/app/certificates/[hash]/page.tsx`

Server component. Calls `VerifyCertificate`, renders a verification page
with:
- Recipient name (or "Anonymous" if revoked? No — always show name)
- Course title + tagline
- Issued date
- Verification hash (truncated with full-on-click)
- Status badge: "VERIFIED" (green) or "REVOKED" (red, with reason)
- "Download PDF" button → `/certificates/{hash}/pdf`

Uses Tailwind classes consistent with the rest of the app. No client
components needed (the page is static + a single link to PDF).

### PDF download — `src/app/certificates/[hash]/pdf/route.ts`

API route (not a page). Calls `RenderCertificatePdf`, returns the
buffer with `Content-Type: application/pdf` and `Content-Disposition:
inline; filename="certificate-{hash}.pdf"`.

Errors:
- 404 if cert not found
- 500 if render fails

The route uses `notFound()` from `next/navigation` for the cert
not-found case so the user sees a nice 404 page.

### 404 — `src/app/certificates/[hash]/not-found.tsx`

Standard Next.js not-found view. Same pattern as the existing
`src/app/courses/[slug]/not-found.tsx`.

## 4. Tests

- `VerifyCertificate.test.ts` — unit tests:
  - happy path (active cert returns full data)
  - revoked cert still returns (with status visible)
  - invalid hash format → invalid_hash_format
  - cert not found → certificate_not_found
  - user not found → user_not_found (defensive)
  - course not found → course_not_found (defensive)
  - db_error on each of the three repos
- `container.test.ts` — wiring test for `verifyCertificate`

**No route tests.** Next.js route testing requires either:
- Playwright (heavy, separate setup, we have it but slower)
- Next's experimental test utilities (not yet stable)
- Mock-heavy unit tests of the route handler (low value)

The page/route logic is so thin (1 use case call + render) that the
use case tests cover the meaningful behavior. The route's only
responsibility is HTTP plumbing, which Playwright covers in a separate
test suite anyway. Deferring route tests to STORY-055 (the
"6 critical-journey E2E tests" story).

## 6. Design Decisions

- **Public routes, no auth** — the hash is the only access control.
  Anyone with the URL can view. This is intentional: certificates are
  meant to be shareable.
- **Hash format validation up front** — saves a DB roundtrip if the
  URL is malformed. Same regex as the entity: `/^[0-9a-f]{64}$/`.
- **HTML view shows revoked status** — but doesn't hide the recipient
  name. The cert still "exists" as a record; the revocation is
  metadata.
- **PDF download works for revoked certs** — consistent with the 042
  decision that the PDF is a historical artifact. The revocation badge
  is the public view's concern, not the PDF's.
- **404 via `notFound()` from next/navigation** — same pattern as
  `/courses/[slug]`. Idiomatic Next.js App Router.
- **No QR code yet** — the URL is already in the PDF footer
  (`/certificates/{hash}`). QR would let you scan it from the printed
  PDF; deferring to a later story (045) since it's a polish item.
